### 学习内容

| Token 经济学 | 缓存策略、Prompt 压缩、Token 计数与成本控制 |

### 🔬 实验

**Lab 1.1**: 构建一个命令行 AI 助手
```
功能需求：
1. 接受用户输入，调用 Claude API 返回结果
2. 支持流式输出（一个字一个字往外蹦）
3. 支持读取本地文件作为上下文（--file 参数）
4. 对话历史保存在本地 JSON 文件里，支持 --continue
5. 处理 API 超时、限流、token 超限等异常

技术点：
- async/await 异步调用
- anthropic Python SDK
- 流式响应的处理
- 文件 I/O + JSON 序列化
```

**Lab 1.2**: 结构化数据提取器
```
需求：
给定一段非结构化文本（如合同、邮件、会议纪要），
用 LLM 提取出结构化信息（人名、日期、金额、关键事项），
以 JSON 格式输出。

技术点：
- 精心设计 System Prompt
- 使用 Tool Use / Function Calling 强制结构化输出
- 批量处理 + 成本统计
```

## Phase 2：RAG 数据管线（5 周）★ 核心阶段

### 为什么这是核心
RAG（检索增强生成）的本质是把**数据工程能力**搬到 AI 场景里：

- 数据采集 → 清洗 → 分块 → 向量化 → 存储 → 检索 → 重排序

你干了 1-3 年的数据开发，ETL、数据质量、Pipeline 监控这些你已经会了。RAG 就是要你把这些能力用在**非结构化数据**上。纯做后端的人做不好这个，因为他们不懂数据工程；纯做算法的人也不爱做这个，因为这本质是工程活。**这就是你的差异化优势**。

### 学习内容

| 周 | 主题 | 具体内容 |
|----|------|---------|
| 1 | 向量化基础 | Embedding 原理、文本→向量、相似度计算(余弦/点积/欧氏) |
| 1 | 向量数据库 | pgvector、Milvus、ChromaDB 的选型和使用 |
| 2 | 文档处理管线 | PDF/HTML/Markdown 解析、分块策略(固定/语义/递归) |
| 2 | 元数据管理 | 文档级别的元数据过滤、HyDE、Multi-Vector |
| 3 | 检索优化 | 混合检索(关键词+语义)、重排序(reranker)、查询改写 |
| 4 | RAG 评估 | RAGAS 框架、检索准确率、生成忠实度 |
| 5 | 高级 RAG | GraphRAG、Agentic RAG、多模态检索 |

### 🔬 实验

**Lab 2.1**: 个人知识库 RAG（1 周）
```
数据源：收集你自己的笔记、PDF、Markdown 文件（至少 50 个文档）

Pipeline：
1. 文档加载器（支持 PDF、Markdown）
2. 递归分块（按段落/标题，每块 500-1000 tokens）
3. 向量化并存入 pgvector
4. 检索接口（输入自然语言，返回最相关文档片段）
5. 调用 LLM 基于检索结果生成回答

技术栈：Python + pgvector + anthropic SDK
关键指标：检索召回率 > 80%，生成回答有明确来源引用
```

**Lab 2.2**: 数据库问答 Agent（2 周）
```
场景：给定一个数据库（可以用你熟悉的任意数据集），
用户用自然语言问问题，系统自动：
1. 判断这是需要查表的问题还是文档类问题
2. 如果需要查表 → AI 生成 SQL → 执行 → 拿到结果 → LLM 解读
3. 如果是文档问题 → 走 RAG 检索

这个实验本质上是 RAG + Text-to-SQL 的结合，
你的 SQL 背景在这里是巨大优势。

技术点：
- Tool Use：定义多个 tool（查表、查文档、计算）
- SQL 安全校验（AI 生成的 SQL 可能有注入风险）
- 多轮对话状态管理
```

**Lab 2.3**: 数据质量驱动的 RAG 优化（2 周）
```
这个实验最贴近你的数据开发背景，也是面试时最能说的项目。

用 RAGAS 框架评估 Lab 2.1 的 RAG 系统，然后系统性地优化：
1. 分块策略对比实验（固定大小 vs. 语义分块）
2. Embedding 模型对比（不同模型对检索效果的影响）
3. 重排序器效果测量
4. Pipeline 监控：记录每次检索的延迟、召回率、用户反馈

输出：一份完整的评估报告，包含数据、图表、优化建议
——这是面试时能让你脱颖而出的东西。
```

### ✅ 检验标准
- [ ] 能解释 Embedding 的原理和相似度计算
- [ ] 能独立搭建从文档采集到问答输出的完整 RAG Pipeline
- [ ] 能用 RAGAS 等工具评估和优化检索质量
- [ ] 能做一个结合 SQL 查询 + 文档检索的混合 Agent

## Phase 3：Agent 开发（4 周）

### 目标
从「一问一答」升级到「让 AI 自主完成多步骤任务」——这是 2026 年最受市场追捧的技能。

### 学习内容

| 周 | 主题 | 具体内容 |
|----|------|---------|
| 1 | Agent 基础模式 | Tool Use / Function Calling、ReAct 模式、Plan-and-Execute |
| 2 | 多 Agent 编排 | Agent 间通信、任务分解与分配、LangGraph |
| 3 | 记忆与状态 | 短期记忆(上下文)、长期记忆(向量库)、工作记忆(scratchpad) |
| 4 | 安全与可控性 | Prompt 注入防护、输出校验、访问控制、Human-in-the-loop |

### 🔬 实验

**Lab 3.1**: 数据分析 Agent
```
结合你的数据开发背景，做一个能自主完成数据分析任务的 Agent：

用户说："分析上周的销售数据，找出同比下降超过20%的品类，
并给我一份简要报告"

Agent 应该：
1. 理解需求 → 拆解为子任务
2. 生成 SQL 查询 → 执行 → 拿到数据
3. 对结果做分析（哪些品类下降、可能的原因）
4. 生成一份 Markdown 格式的分析报告并保存为文件
5. 如果某个步骤失败（比如 SQL 报错），能自行调试修复

技术栈：Claude API + Tool Use + Python + 你的真实/模拟数据库
```

**Lab 3.2**: 多 Agent 协作系统
```
构建一个简单的多 Agent 系统：

角色：
- Planner Agent：分析用户需求，制定计划，分配给 Specialist
- SQL Specialist Agent：专门写 SQL 和查数据库
- Doc Specialist Agent：专门检索文档和报告
- Reporter Agent：汇总结果，生成最终报告

场景：用户提问涉及数据查询 + 业务文档查阅的混合需求

技术栈：LangGraph / 自己的编排逻辑
关键设计：Agent 之间的上下文传递格式
```

### ✅ 检验标准
- [ ] 能手写 ReAct 循环而非依赖框架黑盒
- [ ] 能设计 Agent 的安全边界（哪些操作可以自动执行，哪些需要确认）
- [ ] 能做一个完成多步骤任务的 Agent（不是 demo 级别，是真的能干活）

### 🔬 实验

**Lab 4.1**: 部署一个生产级 RAG 服务
```
将 Lab 2 的 RAG 系统改造成一个可独立部署的服务：

要求：
1. FastAPI 提供 REST API
2. 异步处理、请求队列、超时控制
3. 用 Docker 打包
4. 基本的监控（Prometheus metrics：QPS、延迟、错误率）

6. 健康检查端点

这不是 demo，是能给别人调用的服务
```

**Lab 4.2**: 数据 Pipeline 自动化
```
场景：模拟一个真实场景——每天有新的文档/数据进来，需要自动更新 RAG 系统

Pipeline 设计：
1. 定时检测数据源变化（新增/修改/删除的文档）
2. 增量更新向量库（只重新处理变化的部分）
3. 更新后自动运行评估套件
4. 如果评估指标下降 > 阈值 → 告警 → 人工介入
5. 全流程的日志和监控

技术栈：Apache Airflow / Prefect / 简单的 cron + Python
```

## 备忘：容易踩的坑

1. **不要追新框架**：LangChain、LlamaIndex、DSPy 变化极快，纠结选哪个是浪费时间。先用最裸的 API 写，理解原理后框架只是工具。
2. **不要迷恋微调**：你现在的路线上微调不是重点。RAG + Agent 能解决 80% 的企业需求，而且和你的数据背景匹配。
3. **不要跳过 Linux/Docker**：很多数据分析出身的人觉得这是运维的事就跳过了，结果面试一问三不知。这是工程化的入场券。
4. **所有实验代码放 GitHub 公开**：这是你最好的简历。

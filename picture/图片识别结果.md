## 二、评估体系：衡量改进的基石

### 2.1 评估的必要性
提供严谨性，验证提示词变更是否真正带来性能提升

### 2.2 模型迁移的挑战
新模型可能行为不同（可通过提示词调整）或能力不足（提示词无法弥补）

### 2.3 评估用例设计

| 用例类型 | 说明 |
|---------|------|
| 控制用例 | 模型已能良好处理的明确任务 |
| 边缘用例 | 模型曾失败的场景，通过提示词防止再次发生 |
| 能力边界用例 | 明确模型应转交人工或拒绝回答的请求 |

## 三、案例一：优化现有客服机器人提示词

### 3.1 初始问题与通用清理

**问题**：提示词冗长、结构混乱、包含过时信息（如网站内容）

**清理措施**：
- 使用 XML 标签添加结构（角色、指南、政策、语气等）
- 移除冗余描述

**效果**：初步提升性能，证明结构清晰的重要性

### 3.2 定义输出格式

**最佳实践**：明确输出格式（如使用 XML 标签），并可结合 API 的停止序列确保一致性

**热点数据查询失败**：移除针对旧模型的过度防御性指令（"永远不要给客户错误信息"）

### 3.3 针对性解决失败模式

| 失败模式 | 解决方案 |
|---------|---------|
| 账单计算不准确 | 指令无法增加能力，解决方案是引入计算工具，并在提示词中指导模型何时调用 |
| 账单错误转人工 | 避免单方面指令（"避免转人工，因为成本高"），应提供权衡标准 |

## 四、案例二：从零构建排班智能体

### 4.1 目标
基于员工可用性等约束，生成一周零售员工排班表

### 4.2 评估方式
使用 Python 函数编程检查硬性规则的违反情况

### 4.3 不同方法对比

| 方法 | 结果 |
|------|------|
| 简单模型 + 基础提示词 | 所有用例失败，推理能力不足 |
| 强大模型 + 基础提示词 | 违规次数减少，但仍失败 |
| 强大模型 + 自适应思考 | 可靠生成合规排班，但消耗和延迟大幅增加 |
| 简单模型 + 优化提示词 | 部分成功，但可能因输出限制失败，且优化后延迟可能更高 |

### 4.4 最终方案：智能体工作流

**采用"生成-评估-修复"循环**：
将任务拆分为三个独立提示词——
1. 生成草稿
2. 评估违规
3. 针对性修复

**结果**：成功解决所有用例，代价和延迟更优，并能灵活处理运行时的特性约束

## 五、结论与核心启示

### 5.1 通用清理原则
遵循结构化、清晰化的通用最佳实践，能立即提升评估性能

### 5.2 迭代优化方法
通过评估系统性地定位失败模式，逐一解决：
- 添加结构
- 避免冗长禁令
- 提供工具
- 明确权衡标准

### 5.3 架构选择
对于复杂任务，考虑将**单一复杂提示词拆分为多个专注于特定子任务的提示词系统（智能体工作流）**，以提高可靠性

## 图片内容总结

这是一张关于 **Prompt Engineering 实战方法论** 的思维导图，核心理念是：

> **不要试图用一个大 prompt 解决所有问题。而是：评估先行 → 结构清理 → 拆分任务 → 迭代优化 → 智能体工作流。**

通过两个完整案例（客服机器人优化 + 排班智能体构建），展示了从"单一复杂 prompt"到"生成-评估-修复循环"的演化路径。
